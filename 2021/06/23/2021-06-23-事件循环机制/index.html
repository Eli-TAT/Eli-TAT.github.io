事件循环机制

事件循环机制是一个比较底层且十分重要的概念，有助于更深入地理解浏览器是处理各种任务以及 JS 程序是如何执行的。需要注意的是，在不同的 JS 执行环境中实现事件循环机制的方式略有不同，这里只讨论浏览器环境。node环境下的事件循环机制请参考node官方文档、Node 中的 Event Loop 和浏览器中的有什么区别

介绍事件循环之前需要有一些准备知识。

1. 浏览器中的渲染进程

浏览器是多进程的，渲染进程属于浏览器多进程中的一种，主要负责页面渲染、脚本执行、事件处理等，其中包含的线程有 GUI 渲染线程、JS 引擎线程、事件触发线程等。

GUI 渲染线程

GUI 渲染线程主要用于解析 HTML 文档，对网页进行渲染。
注意：GUI 渲染线程和 JS 引擎线程是互斥的，当 JS 引擎执行时 GUI 线程会被挂起，GUI 更新会被保存在一个队列中等到 JS 引擎空闲时立即被执行。

JS 引擎线程

JS 引擎线程也称为 JS 内核，负责解析和运行 Javascript 脚本。JS 引擎线程会一直等待着任务队列中任务的到来，然后加以处理，一个 Tab 页中始终只有一个 JS 引擎线程在运行 JS 程序；

JS 为什么是单线程的？
js 作为浏览器脚本语言，其主要用途就是与用户交互以及操作 DOM，这决定了它只能是单线程，否则会带来许多同步问题。举个简单的例子，如果有两个线程同时操作了某个 DOM 节点，那么浏览器就不知道该以谁为准了。

但是仅仅依靠单线程是无法处理各种复杂情况的，单线程只能按照顺序执行任务，然而 JS 引擎线程还需要处理那些耗时的任务（如各种 UI 事件、定时器任务、异步网络请求），这样就会造成 JS 引擎线程的阻塞，影响整个页面的性能，这时就需要事件循环机制。

2.事件循环中的概念

ECMAScript中并没有提到事件循环，不过在HTML规范中有详细说明 https://html.spec.whatwg.org/#event-loops

任务队列

在事件循环机制中，任务队列就是一个消息队列，具有队列数据结构的特征（先进先出），用来存放任务（浏览器执行的其他操作，如UI事件的处理函数、定时器任务等），其中任务又分为两类，微任务和宏任务。

微任务和宏任务

  1. 宏任务
    宏任务指的就是浏览器中一些独立的工作单元，主要有：
  • 执行主线（全局） JavaScript 代码
  • setTimeout()、setInterval()
  • UI 交互事件
  • postMessage
  • 网络请求
  • I/O 操作
  • …
  • node 中的 setImmediate()
  1. 微任务
    微任务指的是更小的、实时性较高的任务，用于更新网页应用的状态，需要尽可能快的、通过异步方式执行。微任务使得我们能够在UI渲染之前执行一些操作，避免不必要的UI重绘。
  • Promise的回调then()
  • 监听 Dom 变化的 mutationObsever
  • node 中的 process.nextTick

事件循环的实现至少应该有一个用于宏任务的队列和一个用于微任务的队列。但是由于市面上的浏览器和JavaScript执行环境种类很多，因此具体实现也不太一致。

3. 事件循环过程

注意：单次事件循环中，最多处理一个宏任务（其余的在任务队列中等待），而微任务队列中的微任务都会被处理。可以理解为一个宏任务绑定着一个微任务队列。

  1. 首先执行全局 JavaScript 代码（第一个宏任务），遇到宏任务就推入任务队列（即宏任务队列），遇到微任务就推入微任务队列；
  2. 当执行完当前的一项宏任务后，查询是否有微任务需要执行；
  3. 有则清空微任务队列，按顺序执行微任务，否则直接跳到下一步；
  4. 当执行完所有任务后，如有必要会进行 UI 渲染，至此一轮 Event Loop 结束；
  5. 然后进行下一轮 Event Loop，从任务队列中的第一项宏任务开始执行；

4. 事件循环中的误区

上面所说的遇到宏任务就推入任务队列其实并不严谨，对于异步任务（如事件处理函数、定时器任务）来说，它们并不是直接将回调推入宏任务队列，而是先将任务交给对应的线程进行管理，等到一定时刻才将回调推入任务队列。

  • 当事件监听器被注册后，会由事件触发线程进行管理，一旦监听的事件被触发，才将回调函数推入任务队列。
  • 定时器任务也是如此，设置一个定时器后，它会由定时器触发线程管理，等到定时结束，才将回调函数推入任务队列。这也就可以解释定时器为什么不准了，因为过了设定的延迟时间之后，回调只是被推入任务队列，需要等待前面的任务执行完毕才能执行，并没有立即执行。
