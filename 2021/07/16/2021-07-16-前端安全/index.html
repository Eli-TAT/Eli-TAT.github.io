前端安全

常见的前端安全问题有 SQL 注入、XSS 攻击（跨站脚本攻击）以及 CSRF（跨站请求伪造）。

1. SQL 注入

SQL 注入是指将 SQL 语句插入到 页面请求的查询字符串或 Web 表单中，以达到欺骗服务器执行恶意 SQL 语句，进而对数据库造成破坏。造成 SQL 注入的根本原因在于没有对用户输入的数据进行检验（原则上来说用户的输入都是不安全的），预防措施就是对用户的输入都进行检验、过滤。（前后端均可进行）

2. XSS 攻击

XSS（cross site script）全称跨站脚本攻击，指的是攻击者在网页中插入恶意脚本，当用户浏览该网页时嵌入的脚本就会被执行，从而达到恶意攻击的目的。

XSS 攻击原理

  1. 恶意用户想方设法在页面中插入一段脚本，比如在论坛留言区丢入一段 script 脚本。
  1. 浏览器在渲染 HTML 文档时会直接执行 script 标签中的代码，并不会因为 script 标签出现在留言区而察觉到异常。除此之外，script 标签还能嵌入到页面请求的查询字符串中。
  1. 并不是只有 script 标签中才能插入脚本，img、a 等标签内的事件属性也能直接执行 js 代码。

XSS 攻击分类

  1. 反射型 XSS

反射型 XSS，又称非持久性 XSS，也就是攻击相对于访问者是一次性的，服务器只是将恶意脚本不加处理的“反射”给访问者的浏览器进行执行。上面所说的在页面请求的查询参数中嵌入脚本就属于这一类型，只有在访问特定的 url 时才会触发。

  1. 储存型 XSS

储存型 XSS，又称持久型 XSS，也就是攻击相对于访问者是持续的，由于恶意脚本被存储在数据库中，服务器会将恶意脚本当做数据嵌入页面，任何用户只要访问该页面就会遭受到攻击，因此它的危害是 XSS 攻击中最大的。例如上面所说的在论坛留言板插入脚本。

  1. DOM XSS

顾名思义，DOM 跨站脚本攻击是指网页在 DOM 操作过程中造成恶意脚本被执行，用户因而遭到攻击，在这个过程中服务器响应的页面内容并没有发生改变，一切发生在本地环境中。

如何防范 XSS 攻击

  • 首先是过滤。在输入时对诸如 script、img、a 等标签进行过滤。
  • 其次是编码。一些常见的符号，如<>在输入的时候要对其进行转换编码，这样浏览器则不会对其进行解释执行，同时也会不影响显示效果。
  • 最后是限制。xss 攻击往往需要构造较长的字符串，因此对于一些可以预期的输入可以通过限制长度强制截断来进行防御。

对于一些 SPA（单页面应用）框架，事先替开发者做好了对 XSS 攻击的预防处理。除此之外，浏览器方面对 XSS 的防范措施也越来越严格。总而言之，防范 XSS 攻击并不困难，只需要开发者考虑足够细致。

3. CSRF 攻击

CSRF（Cross-site request forgery）全称跨站请求伪造，简单来说就是通过盗取 cookie 等方式伪造用户身份，向服务器发起恶意请求。例如盗取某银行用户的登录 cookie 后，就可以伪造其身份向服务器发起取款、转账等请求。

如何防范 CSRF

防御 CSRF 攻击主要有三种策略

  1. 验证 HTTP Referer 字段

在 HTTP 请求报文头部有一个字段叫 Referer，它记录了该 HTTP 请求的来源地址。由于 CSRF 一般是在攻击者自己所在的域进行，利用该字段，开发者可以在服务器接收 HTTP 请求时对其来源地址进行验证，如果来源地址符合规定，那么请求就是合法的，否则就有可能是 CSRF 攻击，拒绝该请求。

设置 Referer 字段并不是绝对安全的，在一些旧版本浏览器（比如 IE6 或 FF2）中存在可以修改 Referer 字段的 API，攻击者可以借助修改 HTTP 请求头中的 Referer 字段来达成 CSRF 攻击。即使在高版本浏览器中，由于 Referer 字段会记录下用户的访问来源，可能涉及用户隐私，所以用户可以自行设置请求时是否携带 Referer 字段，一旦用户选择不携带，合法用户的请求也会被认定为非法。

  1. 在请求地址中添加 token 并验证

也就是在请求参数中加入 token 进行验证。

  1. 在 HTTP 头中自定义属性并验证

相较上面每个请求地址中都加入 token，在 HTTP 头部中添加自定义属性携带 token 显得更为便捷，通过 XMLHttpRequest 对象，可以为所有该类请求加上自定义属性携带 token 进行验证。
