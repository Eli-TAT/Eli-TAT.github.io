总结一些前端加密的知识。

前端加密的意义
和后端加密一样，前端加密的意义同样在于保护数据的安全，但是由于前端的透明性给了非法用户可乘之机，导致了前端加密的安全性不足，往往需要配合后端加密等手段，才能保证较高的安全等级。

前端加密方式

  • 使用 js 将数据加密后进行传输
  • 使用 HTTPS 传输数据
  • 浏览器插件内进行数据加密后传输

1. js 加密

js 对数据的加密可以通过各种加密算法实现，下面将对它们做简单的介绍。

（1）对称加密

对称加密是最简单快捷的一种加密方式，它使用同一个密钥对数据进行加密和解密。对称加密通常使用较小的密钥，一般小于 256bit，密钥的大小直接关系着加密和解密耗费的时间，因此既要考虑安全性也要考虑效率。

  • 优点：方便使用、速度较快；
  • 缺点：安全性不高，一旦密钥泄露，加密后的数据很容易就会被破解；

常用的对称加密算法有和 AES 和 DES 等。

登录密码等敏感信息不要仅仅在前段使用 js 进行对称加密，因为其他人一旦获取到保存在前端的密钥，可以直接对密文进行解密。对称加密的一大问题是密钥的管理与分配，即如何将密钥传送到需要的人手中，针对密钥可能被黑客获取或拦截的情况，通常做法是将对称加密的密钥进行非对称加密，然后发送给需要的一方。

（2）非对称加密

非对称加密提供了一种十分安全的加密与解密的方法，他使用一对密钥（公钥和私钥）进行数据的加密和解密。其中私钥只能由一方安全保管（通常是服务端）不能泄漏，公钥则可以提供给任何人。例如前端向服务器请求公钥，服务器将公钥发送给前端后，前端通过该公钥对数据进行加密后发送给服务器，只有持有私钥的服务器才能将密文解密，从而大大提高了安全性。

  • 优点：安全性高，无需将密钥暴露出去。
  • 缺点：速度相较对称加密慢很多，影响效率。

常用的非对称加密算法有和 RSA 和 DSA（数字签名）等。

由于非对称加密速度较慢，所以还是应该使用对称加密传送消息，可以考虑通过非对称加密的方式发送对称加密所需的密钥，这样就可以很好的解决安全性和效率的问题。

（3）哈希算法

哈希（Hash）是指将目标文本转换为具有固定长度的字符串（即消息摘要），哈希算法指的就是具有这类特性的一类算法。哈希的过程是不可逆的，且原始数据和哈希后的数据呈现多对一的映射关系，也就是说一个原始数据经过某个哈希算法处理后只有一个哈希值，但不同的原始数据可能对应着同样的哈希值。

哈希算法适用于被保护数据仅用作比较验证且不需要还原成明文形式，常见的例子就是，当需要重置账户密码时，网站会给用户一个邮箱激活链接或者是一个随机的密码，而不是将原始密码发给用户，因为密码经过哈希之后是不可逆的。

需要注意的是，我们不仅仅要在前端使用哈希算法，服务端也需要进行哈希，这样既能避免需要将前端发来的密文解密成明文再和数据库中的数据进行对比，也避免了数据库泄露之后密码相当于是明文存储的情况。

常用的哈希算法有 MD5、SHA1 等。

对于哈希算法的攻击方式主要有：寻找碰撞法和穷举法。为了保证数据的安全，可以在哈希算法的基础上做进一步的加密，常见方法有：加盐、慢哈希、密钥哈希、XOR 等。 更多相关知识可参考

（4）HMAC

HMAC 全称哈希运算消息认证码，它是一种基于 Hash 算法和密钥进行消息认证的方法，可以与任何哈希算法（MD5、SHA256 等）配合使用。它主要用于身份验证，用法如下：

  1. 客户端发起请求（第一次请求）
  2. 服务端返回随机码
  3. 客户端将随机码作为密钥对登录密码进行加密，将密文发送给服务端（第二次请求）
  4. 服务端从数据库中读取用户密码，用密钥进行和客户端一样的 HMAC 运算，与用户发送的结果进行比较，如果一致则验证成功，请求结束随机码失效

使用 HMAC 的好处在于，既可以加密传输数据（防止黑客直接截取 HTTP 报文），也可以防止重放（防止黑客截取登录请求报文后再次发送给服务端）。如果黑客截取第一次请求获取到了随机码，不知道密码仍然无法登录；如果被截取的是第二次请求，由于第二次请求结束随机码就会失效，即使将请求重放黑客也无法验证成功。

（6）Base64 编码

Base64 是一种基于 64 个可打印字符来表示二进制数据的表示方法，常用于表示、传输、存储一些二进制数据。它只是一种通过查表的编码方法，不能用于加密。

Base64 适用于小段内容的编码，比如数字证书签名、Cookie 的内容等，目前前端中 Base64 用得最多的应该就是图片转码。

（7）数字签名

数字签名主要用于确认信息来自于特定的主体且信息完整未被篡改。其中数据发送方生成签名，接收方验证签名。
